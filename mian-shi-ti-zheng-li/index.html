性能优化

开屏时间优化

  1. 通过延迟加载内部框架，将一个占用 40% 启动时间的库从 application:didFinishLaunching 中剥离，放到首页渲染后再异步初始化，使冷启动时间减少了近 30%
  2. 利用 Apple 在 WWDC19 推出的 App Launch Instrument，分析发现主线程被 Storyboard 反序列化和资源加载阻塞。我们将 LaunchScreen 简化为纯色背景+Logo，减少节点数，并将推送注册、统计上报等非关键逻辑延迟到后台线程执行，启动渲染速度提升了约 25%
  3. 联合后台减少首屏接口的数据量，并借助阿里云对所需的图片进行裁切和压缩，进一步降低 I/O 开销

安装包体积优化

  1. 统一将图片资源转换为 HEIF 格式，并利用 Asset Catalog 压缩功能，单张图片体积平均减少约 40%。部分不支持 HEIF 的场景下，确保 png 为 8 位格式。
  2. 在 build setting 中开启一些优化项，可以去除未引用的符号和多余架构。
  3. 可以将大体积的图片和视频资源，或者主题和皮肤这些不常用的资源迁移到 ODR，使用时按需加载。
  4. Bitcode 现在已经被 apple 弃用了。

列表性能优化

  1. 除了常见的比如少减少列表中的离屏渲染；在后台线程预先计算 Cell 高度；减少 Cell 内视图层级；减少对 UIKit 属性的操作，比如对 frame 的操作；使用隐藏和显示来代替子视图的增加和删除；这些常规优化操作之外，如果项目最低支持是 iOS 13，那么我们还可以引入 diff 算法。

锁

操作系统的锁有哪几种

  1. 互斥锁：@synchronized，NSLock，还有一些更底层的 api
  2. 条件锁，NSCondition：允许线程在特定条件满足时才继续执行
  3. 读写锁：GCD提供的栅栏函数，支持多读单写
  4. 递归锁和自旋锁
  5. 信号量 和 dispatch_once，严格意义来说不是锁，但是能起到锁的效果

OC如何实现类似Swift的异步函数的同步实现（不考虑性能问题）

  1. 使用 GCD 的信号量或者 NSCondition 去卡线程和异步回调。不过需要注意线程问题，否则有可能导致界面假死甚至死锁。
  2. 使用 GCD 的 group，用 enter、leave 和 wait 这三个 api 来控制。
  3. 使用 NSOperationQueue 设置子任务之间的依赖性，达到顺序执行的目的
  4. 还可以在执行异步操作之后，用 while + RunLoop 的 runMode beforeDate api 做一个轮询。

Swift 协程

actor

actor 是 Swift 协程提供的引用类型，内部状态通过串行化执行队列保护，保证同一时刻只有一个执行上下文访问其可变属性，进而实现属性的的线程安全。

JSBridge

JSBridge 基本就是对 evaluateJavaScript 和 window.webkit.messageHandlers.<handler>.postMessage() 的封装，内部用一个 callbackId + 闭包的字典存回调。

可以通过拦截 prompt 对话框的方式实现 js 同步获取到 Native 的返回值。因为 web 端会等待方法的返回。
缺点：

  1. 非常规用途，在某些版本的 iOS / Safari 可能弹出警告窗口或行为变化
  2. JS 可以随便调 prompt，如果没做好校验可能被滥用
  3. 只能传字符串，但是 postMessage 在 iOS 这边可以传一些复杂数据结构，比如对象，在 iOS 这边会被映射为字典。

另外理论上还可以通过拦截 url 重定向来同步返回结果。

CoreText

图片混排

可以添加一个占位符，然后通过 CTRunDelegate 设置这个占位符的大小和位置，最后用 drawRect 把这个图片画在这个位置上就行。

点击交互

可以在 touchesBegan 方法里获取到点击坐标，然后把这个坐标转换为 CoreText 的坐标，因为 UIKit 的坐标系原点在左上角，但是 CoreText 的原点是在左下角的。
然后可以用 GetLines 和 GetStringIndexForPosition 获取到点击的字符串的索引。最后判断是否在可点击的文本范围内。

GCD 的核心概念

GCD 是一种基于队列的并发编程技术，主要用于管理任务的提交和执行。
GCD 主要使用任务和队列的概念，任务是需要执行的代码块，队列决定如何执行

同步：当前任务必须等待同步代码块执行完毕后才能继续，会阻塞当前线程
异步：提交任务后立即返回，不会阻塞当前线程，任务可能会在后台执行

串行队列：任务按照添加顺序依次执行，无论同步还是异步，都是按顺序执行
并发队列：任务可以并发执行，异步时开启多个线程执行任务，同步则仍然会阻塞当前线程，不会开启新线程

iOS 应用的内存布局

栈：存放局部变量，函数调用信息。由系统自动管理，访问速度快，空间较小，数据生命周期短。
堆：动态分配的对象，闭包捕获的变量。由开发者手动管理，访问速度慢，空间更大，数据生命周期长

class Person {
    var name: String
    init(name: String) {
        self.name = name
    }
}

let p = Person(name: "John") // p 存储在栈上，但 Person 实例存储在堆上

全局/静态存储区：全局变量，静态（static）变量。程序启动时分配，进程结束才释放。
常量区：只读数据，例如 let 声明的变量，Swift 编译优化后的静态常量。访问速度快，因为只读所以不可修改。
代码区：程序的可执行代码，函数的二进制指令。只读，防止恶意篡改，由系统加载到内存，执行时直接读取。

ObjC 方法查找流程

  1. 方法缓存查找：如果缓存中有方法实现，则直接执行
  2. 方法列表中查找：如果缓存未命中，则从类的方法列表中查找
  3. 沿继承链向上查找：如果当前类找不到，则沿着 superclass 继续查找
  4. 动态方法解析：如果方法未实现，可以通过动态方法机洗添加方法
  5. 消息转发：如果解析失败，ObjC 允许消息转发给其他对象处理
  6. 最终触发崩溃

RunLoop

RunLoop 是事件驱动的循环，本质是一个事件处理机制，它的作用是 管理线程的生命周期，并处理输入员和定时器等事件。
每个线程都有一个 RunLoop，只有主线程的 RunLoop 默认是开启的，子线程需要手动开启 RunLoop。每个线程最多只有一个 RunLoop。

执行流程

  1. 进入 RunLoop，获取当前模式
  2. 检查当前模式下是否有带处理的事件
  3. 如果有事件，处理事件后继续循环
  4. 如果没有事件，则让线程进入休眠状态，等待新的事件到来
  5. 如果 RunLoop 退出，则线程结束或者 RunLoop 停止运行

应用常见

  1. 可以通过 RunLoop + NSMachPort 实现线程保活。比如后台任务线程，Socket 连接线程。
  2. 通过 DispatchQueue.main.async 将任务提交到主线程 RunLoop，以实现在 RunLoop 空闲时执行任务的功能，比如空闲时再绘制 UI。
  3. 在子线程调用 perform(_:with:afterDelay:) 方法时，需要手动开启 RunLoop 确保任务执行。
